# Phase 3 — Buyers, Documents, Invoices (A) ∥ Invoice Wizard (B)

**Objective:** a supplier submits a complete invoice that reaches `ELIGIBLE`, with OCR/QR extraction genuinely running, duplicate detection genuinely blocking, and the buyer resolved — never auto-selected — into the global directory.

## Agent A tasks

- [ ] Buyer search `/buyers/search`: own relationships → platform buyers → CCD adapter; returns candidates + `requiresManualReview`; **never auto-selects** (ZM-BUY-009).
- [ ] `/buyers/resolve`: global dedup on national number; create/link `Buyer` + `SupplierBuyerRelationship` (contact data on the relationship, never on `Buyer` — ZM-BUY-005/008); 409 on SUSPENDED/STRUCK_OFF; `UNDER_LIQUIDATION` → manual review (LT-02 policy).
- [ ] `buyer_resolution_attempts` recorded; `BUYER_MANUAL_REVIEW` path.
- [ ] Documents: `upload-url` / `download-url` with short-lived signed URLs issued **after server-side authorization** (ZM-DOC-004); content hash, MIME, size recorded.
- [ ] Python service (in `/services/ml`): OCR structured extraction for the seeded e-invoice formats + local QR decode — schema-driven, tolerant, degrades to `UNPARSED` (ZM-DOC-010); dummy e-invoice validation adapter (ZM-DOC-009); write `docs/specs/EINVOICE_QR.md`.
- [ ] `document_extractions`: raw output preserved separately from supplier corrections; both retrievable via `/documents/{id}/extraction` (ZM-DOC-006).
- [ ] Transactions: create draft, invoice PUT (server recomputes `outstandingAmount`), buyer link PUT, `minimum-amount` PUT (validated ≤ outstanding, > 0), declarations POST (all-true enforced), submit.
- [ ] Fingerprint generation (parties + invoice number + date + value + tax) + duplicate detection per D-01 mechanism; collision → 409 + review record (ZM-VER-001).
- [ ] Verification runs: all 8 check types (completeness, identity, duplicate, logic, eligibility, file integrity, OCR consistency, QR consistency) with recorded results; pipeline `SUBMITTED → AUTOMATED_CHECKS → ELIGIBLE | UNDER_REVIEW | FRAUD_REVIEW`; AS-07/AS-08 (past-due ineligible; min tenor 7 days).
- [ ] Snapshot-freshness trigger: re-query government when stale on new submission (ZM-GOV-005).
- [ ] Seed: e-invoice PDFs **generated by us** (known layout + QR) incl. one with a deliberate OCR-vs-entered mismatch; a duplicate pair across two suppliers.

## Agent B tasks

- [ ] Invoice wizard step 1 — Buyer: search → candidate list (name, type, national number, status, governorate) → **explicit selection, never pre-selected even at 100% match** → contact form (name, role, phone, email, branch); SUSPENDED/STRUCK_OFF blocked with clear reason; manual-review path surfaced.
- [ ] Step 2 — Invoice: e-invoice upload → OCR pre-fill → extracted-vs-entered comparison with mismatch highlighting; corrections **recorded alongside, not replacing** extraction (explicit copy in UI); QR status shown incl. `UNPARSED` → manual-review note.
- [ ] Step 3 — Documents: additional uploads per policy (PO, delivery note, statement).
- [ ] Step 4 — Minimum acceptable amount: labelled **minimum NET amount you will receive**; privacy note "banks never see this".
- [ ] Step 5 — Declarations: all eight, all required, template version shown.
- [ ] Step 6 — Review & submit; duplicate-fingerprint 409 rendered as a clear blocked screen with review reference.
- [ ] Supplier transaction list + detail (states through `ELIGIBLE`); verification results panel.

### Screens in scope (B)

## Ownership & collision guard

## Dependencies

## Integration checkpoint

Live: supplier searches buyer → resolves → uploads seeded e-invoice → real OCR pre-fill with the seeded mismatch highlighted → corrects it (both values retained and retrievable) → sets floor → declares → submits → transaction reaches `ELIGIBLE`. Re-submitting the same invoice from the second seeded supplier is blocked by fingerprint with a review record. Signed-URL authz drill: a bank JWT cannot fetch the supplier's document.

## Definition of done
