## Part 0 — Standing assumptions and judgement calls

These are decisions I made where the pack is silent. Each is flagged so the product owner can override. They are all reversible at low cost if overridden before the phase that consumes them.

| # | Assumption | Basis | Override cost |
|---|---|---|---|
| PA-01 | **Bank onboarding is seed-only in V3.** No API endpoints exist for the bank `INVITED → ACTIVE` lifecycle and none are added. The 3 demo banks are created `ACTIVE` by seed with real user accounts. | Requirements §4.1 "bank onboarding with demo organizations and real user accounts"; no endpoints in the contract | Low before Phase 9 |
| PA-02 | **User/role management is seed-only in V3.** "Bank Admin manages bank users" (ZM-IAM roles) has no contract endpoints; all users and role grants come from seed. Admin screens show memberships read-only. | No endpoints in contract; competition scope | Medium — would need a contract amendment |
| PA-03 | **Reports (ZM-AUD-008) ship as read-only dashboard aggregates computed client-side from existing list endpoints**, not as dedicated report endpoints. Confidentiality rules are inherited from the underlying endpoints. | No report endpoints in contract | Low; additive endpoints possible later |
| PA-04 | **Registration flow:** account signup, email verification, and phone verification happen client-side against Supabase Auth directly (Agent B), not through the NestJS API. The NestJS `users` row is created lazily on first authenticated call. Organization bootstrap is defect D-04 (needs ruling). | `00_START_HERE` §8: Supabase Auth; no auth-registration endpoints in the contract | Low |
| PA-05 | **All background time logic goes through an injected `TimeProvider`** (`now()` = wall clock + `demo_time_offsets` in demo mode, wall clock only in production). This is how the time machine works without touching domain logic. Non-negotiable design rule for Agent A from day one — retrofitting it is expensive. | ZM-DEMO-003/004 | High if deferred |
| PA-06 | **Post-funding outstanding balance is derived from `buyer_payments`**, never by mutating `invoices.paid_amount`/`outstanding_amount` (whose CHECK constraints forbid a fully-paid row — see D-13). The demo itself runs on the deployed stack with the local stack as fallback. | ZM-ARC-001; §8 of START_HERE | Low |
| PA-08 | **The Python ML service trains on clearly-labelled synthetic data generated by a seeded script** (deterministic), producing a small gradient-boosting or logistic model with feature attributions. | PA-10 | **Effort unit:** 1 agent-day = one focused working session of one agent. Ranges below assume the contract-defect rulings arrive on time; add ~20% otherwise. | — | — |

Ordering is by dependency. "A" = Agent A (backend), "B" = Agent B (frontend). Every phase ends with an **integration checkpoint** that is demonstrable, not aspirational. Phases 2–8 genuinely parallelize because B builds against MSW mocks generated from the frozen contract and swaps endpoints to live as A announces them; the checkpoint of each phase is the moment the swap is proven for that phase's endpoints. A and B may sit in different phase numbers at once (B builds ahead on mocks); checkpoints are the only synchronization points.

### Phase 0 — Rulings, scaffold, and ground rules (product owner + both agents)

- **Objective:** remove the known landmines before either agent writes feature code.
- **Product owner:** rule on Part 7 defects D-01..D-05 (minimum: D-01, D-02 before A starts; D-03..D-12 before Phase 2). Approve contract amendment v3.1.0 (additive endpoints) and migration `0002` (additive tables/grants) if accepted.
- **A builds:** monorepo scaffold (`/apps/api`, `/services/ml`, `/db`, `/apps/web` placeholder), CI (lint + typecheck + test on PR), Supabase project, `.env` conventions, migration runner.
- **B builds:** nothing yet (not started, or reads the pack).
- **Dependencies:** none.
- **Integration checkpoint:** repo on `main` with green CI; `docs/coordination/` exists with `DAILY_LOG.md` and `OPEN_QUESTIONS.md`; defect rulings recorded in `DECISIONS.md`.
- **Definition of done:** both kickoff prompts can be pasted and neither agent's first day hits an unresolved defect.
- **Effort:** 0.5–1 day (mostly the owner's rulings).

### Phase 1 — Foundation (A) ∥ Shell (B) — *the only genuinely serial phase*

- **Objective:** auth, organizations, multi-org context, RLS, audit — built once, by A. Nothing downstream starts until `/auth/me` works end to end.
- **A builds:**
  - Migration `0001` = frozen schema (with the D-01 fix as ruled) + `0002` additive (as ruled).
  - Supabase Auth JWT validation; `users` row sync (PA-04); `X-Organization-Id` context guard; membership + role checks; 403 semantics exactly per contract rule 1.
  - RLS: helper functions from the schema, policies for **every** tenant table (the schema shows the pattern for 8 tables; A completes the set), plus the D-02 column-privilege fix.
  - Audit interceptor (every mutation → `audit_logs` with actor user + actor org + correlation id), error envelope, structured logging, correlation-ID propagation.
  - `GET /health`, `/auth/me`, `POST /auth/context`, `PATCH /auth/language`.
  - `TimeProvider` (PA-05) wired into the DI container from the start.
  - Dev seed: 1 user in each of the three org types so B can log in as all personas.
- **B builds (in parallel, no backend dependency):**
  - Next.js app router with `[locale]` segment; EN + AR message catalogs; full RTL layout plumbing (`dir`, logical CSS properties, mirrored nav); **no locale auto-detection** (ZM-I18N-003).
  - Design system primitives: colors, type, spacing, Button, Input, Table, Modal, Toast, Badge, Tabs, Skeleton; `MoneyDisplay`/`MoneyInput` on a decimal library with a lint rule banning `parseFloat`/number arithmetic on money.
  - OpenAPI codegen pipeline: typed client + MSW mock handlers generated from `03_API_CONTRACT.yaml`; per-endpoint mock/live switch map (see Part 3).
  - Supabase Auth login/register/verify UI; org-context switcher in the header (mocked until checkpoint).
  - Role-gated navigation shells for supplier / bank / platform portals; empty/loading/error state patterns.
- **Dependencies:** Phase 0 rulings D-01/D-02.
- **Integration checkpoint:** a seeded user logs into the deployed (or local) stack through the real UI, `/auth/me` returns live memberships, the org switcher changes context via `POST /auth/context`, the language toggle persists via `PATCH /auth/language`, and an RLS smoke test proves a supplier JWT cannot read another org's rows by direct SQL.
- **Definition of done:** checkpoint passes; audit rows appear for the context switch; CI runs the first RLS test.
- **Effort:** A 4–6 days; B 4–6 days (same calendar window).

### Phase 2 — Onboarding + government (A) ∥ Onboarding UI (B)

- **Objective:** a supplier can be taken from registration to an APPROVED, ACTIVE organization by a platform reviewer, with the SLA clock provably pausing and resuming.
- **A builds:** applications CRUD + state machine; SLA clock with business calendar (Sun–Thu 08:00–17:00 Asia/Amman + holidays table) and `sla_clock_events`; government adapter interface + CCD/ISTD/GAM dummy adapters (deterministic seeded data keyed by establishment number, full/partial/unavailable variants, latency + failure injection — per the Government Dummy Data Spec, Part 2); snapshot persistence + 90-day freshness; `entity_field_values` provenance; consents; information requests + respond (SLA pause/resume); reviewer decision endpoint; org bootstrap per D-04 ruling; amendment endpoints `GET /onboarding/applications` (queue + own).
- **B builds:** registration + verification (PA-04); onboarding wizard (establishment number → licence → consents → bank account); government-derived fields **read-only with source badge + retrieval date**; SLA tracker with paused-state reason; information-request inbox + response form; platform review queue + decision screen. All against mocks first.
- **Dependencies:** Phase 1 (auth/context); D-04 ruling; `GET /onboarding/applications` from v3.1.0.
- **Integration checkpoint:** live end to end: register → wizard → submit (SLA starts) → reviewer requests information (clock pauses, supplier sees paused state + reason) → supplier responds (clock resumes) → reviewer approves → supplier org becomes ACTIVE. Government fields render with CCD badge from a real dummy-adapter snapshot. Kill the GAM dummy (failure injection): application enters `GOVERNMENT_SERVICE_UNAVAILABLE`, clock paused, nothing rendered as adverse.
- **Definition of done:** checkpoint + unit tests on business-time arithmetic (holiday spanning, pause/resume reconstruction from events) + `sourceAvailable=false` path tested.
- **Effort:** A 5–7 days; B 4–6 days.

### Phase 3 — Buyers, documents, invoices (A) ∥ Invoice wizard (B)

- **Objective:** a supplier can submit a complete invoice that reaches `ELIGIBLE`, with OCR/QR extraction genuinely running and duplicate detection genuinely blocking.
- **A builds:** buyer search (own → platform → registry; candidates only, **never auto-select**); `POST /buyers/resolve` with global dedup on national number + `SupplierBuyerRelationship`; blocked-status 409s; documents upload/download signed URLs with server-side authz; Python FastAPI service: OCR (structured field extraction from the seeded e-invoice formats) + local QR decode (schema-driven, degrades to `UNPARSED` — ZM-DOC-010); `document_extractions` with raw output preserved separately from corrections; invoice CRUD; fingerprint generation + duplicate detection (per D-01 ruling); declarations; verification runs (all 8 check types); transaction submit pipeline `SUBMITTED → AUTOMATED_CHECKS → … → ELIGIBLE | UNDER_REVIEW | FRAUD_REVIEW`.
- **B builds:** the six-step invoice wizard exactly per brief §Phase 3 — buyer search/candidate list with explicit selection and SUSPENDED/STRUCK_OFF blocks; upload + OCR pre-fill with extracted-vs-entered mismatch highlighting ("corrections recorded alongside, not replacing"); documents step; minimum-amount step labelled **minimum NET amount** with the privacy note; declarations (all required); review + submit; duplicate-fingerprint 409 surfaced as a clear screen.
- **Dependencies:** Phase 2 (an ACTIVE supplier); ML service skeleton (extraction only — scoring comes in Phase 4).
- **Integration checkpoint:** live: supplier searches a buyer, resolves it, uploads the seeded e-invoice PDF, sees real OCR pre-fill with one seeded mismatch highlighted, corrects it, sets the floor, declares, submits, and the transaction reaches `ELIGIBLE`. Submitting the same invoice again from a second seeded supplier is blocked by fingerprint with a review record.
- **Definition of done:** checkpoint + extraction raw/corrected both retrievable via `/documents/{id}/extraction` + signed-URL authz test (bank JWT cannot fetch a supplier document).
- **Effort:** A 6–8 days (Python service included); B 5–7 days.

### Phase 4 — Risk and ML (A; B light)

- **Objective:** every `ELIGIBLE` transaction carries a real, versioned, explainable Trust Score whose availability measure is provably independent of the score.
- **A builds:** deterministic rules engine (hard blockers ML can never override — ZM-RSK-015); synthetic training data generator + training script with recorded metrics; inference endpoint in the ML service; `RiskModelVersion` lifecycle (create/activate, never edit — ZM-RSK-009..011); composite + five components + `dataAvailabilityPct`; reason codes, positive/risk factors; rules-only fallback with visible flag when the ML service is down; `GET /transactions/{id}/risk`; admin `/admin/risk-models`.
- **B builds:** `TrustScoreGauge`, `ComponentBars`, `FactorList`, availability shown **separately and neutrally** (no warning colors, no downward arrow — brief §5), disclaimer text both languages, model version + date + `mlUsed`/fallback flag display. Mostly mock-driven; B's main calendar time this phase continues Phase 5 screens.
- **Dependencies:** Phase 3 (real transactions to score).
- **Integration checkpoint:** live score on an eligible transaction with explanation factors; stop the ML container → recompute → rules-only score with visible degraded flag; run the ZM-RSK-005 test — mark a government source unavailable, recompute, **components unchanged, availability lower**.
- **Definition of done:** checkpoint + INV-9 automated test + historical score immutability test across a version activation.
- **Effort:** A 4–6 days; B 1–2 days (components only).

### Phase 5 — Marketplace + offers (A) ∥ Marketplace/offer UI (B)

- **Objective:** the confidential marketplace works: listing with fee, filtered eligibility, maker/approver offers, and a supplier comparison that leaks nothing.
- **A builds:** listing activation (creates listing-fee obligation + ledger receivable, evaluates eligibility for every active bank with recorded rules — ZM-MKT-003, notifies eligible banks, opens window); deadline jobs (auto-close at `offerSubmissionDeadline`, reminders at 50%/15%, expiry at selection deadline → back to `ELIGIBLE`); policy filter CRUD (+ v3.1.0 PATCH/DELETE); offer create/revise/withdraw with server-side net recomputation and commission/listing-fee injection; maker/approver flow (`PENDING_INTERNAL_APPROVAL → ACTIVE`, self-approval 403 at service **and** DB constraint); floor check with the generic `OFFER_BELOW_SUPPLIER_REQUIREMENT` rejection; **confidentiality enforcement** — response serializers with explicit allow-lists for bank-facing payloads, `offerCount` supplier-only, `/listings/{id}/offers` role-split behaviour; v3.1.0 additions (`GET /offers` list for approval queue/my-offers, `GET /marketplace/listings/{id}` single bank view, `GET /transactions/{id}/listing`).
- **B builds:** supplier listing-activation screen with **fee shown before confirmation** + "applies whether or not financing succeeds" warning; bank marketplace feed; bank underwriting view (identity, invoice, documents, Trust Score block from Phase 4, double-financing disclaimer per ZM-CON-018); offer creation form (gross + bank deductions; commission and listing fee server-computed read-only; net previewed live but always server-verified); approval queue with creator shown and self-approval blocked in UI too; policy filter configuration screen; **supplier offer comparison** — the most important screen: full per-offer breakdown, net payout as anchor, transaction/recourse type with plain-language explanations, conditions with mandatory flags, countdown, **no amount sort, no "best" badge**.
- **Dependencies:** Phase 4 (score shown to banks); v3.1.0 endpoints.
- **Integration checkpoint:** live: supplier activates a listing (fee obligation visible in ledger); bank A's maker creates an offer, approver approves it (self-approval rejected when tried); bank B does the same; supplier sees both offers fully; **bank A's session shows only its own offer and no competitor count — verified three ways: UI, raw API response diff, and direct SQL under bank A's JWT (RLS)**; an offer whose net is below the floor is rejected with the generic message and the response body is byte-scanned for the floor value.
- **Definition of done:** checkpoint + INV-8 serializer tests + INV-11 RLS tests + eligibility evaluation records queryable.
- **Effort:** A 6–8 days; B 6–8 days.

### Phase 6 — Selection + contracts (A) ∥ Acceptance + signing UI (B)

- **Objective:** atomic acceptance and a signed contract.
- **A builds:** `POST /offers/{id}/accept` exactly per brief §5 (single transaction, `FOR UPDATE`, re-validations, lock, SELECTED/NOT_SELECTED, snapshot with hash, audit) + concurrency test harness; `reject-all`; selection-deadline expiry; contract template engine (versioned templates per `transactionType` + fallback, EN/AR, merge fields from snapshot + verified party data); pre-contract checks (ZM-CON-006); dummy signature provider (click-to-accept capturing signer identity/capacity/IP/device/hash); `SignatureVerification`; `FULLY_SIGNED` transition; condition fulfilment endpoints.
- **B builds:** acceptance confirmation modal spelling out atomicity and irreversibility; post-acceptance state screens; contract review + click-to-accept signing; per-party signature status; conditions checklist with fulfilment.
- **Dependencies:** Phase 5.
- **Integration checkpoint:** live: supplier accepts the **lower** of two offers (proving no best-offer logic interferes); the other bank's offer flips to `NOT_SELECTED` and that bank is notified without learning anything else; the concurrency test (two parallel accepts, 20 iterations) yields exactly one winner + one clean 409 every time; both signatories sign; contract shows `FULLY_SIGNED` with hash.
- **Definition of done:** checkpoint + INV-1/2/3/4 tests green in CI + snapshot immutability test (revise source offer after acceptance → snapshot unchanged).
- **Effort:** A 4–6 days; B 3–4 days.

### Phase 7 — Funding, OTP, settlement, ledger (A) ∥ Funding UI (B)

- **Objective:** cross-party funding confirmation and money-correct settlement.
- **A builds:** mark-sent (idempotent, → `FUNDING_CONFIRMATION_PENDING`); OTP generation (hash-only storage, 15 min / 5 attempts / 3 resends from settings, full `funding_otp_events` audit, plaintext returned once to the bank user); OTP verification with generic failures + `attemptsRemaining`; **`FUNDED` requires OTP verified AND settlement evidence** (INV-10); stalled-confirmation escalation job (24h → Ops Admin task, AS-04); settlement adapter (dummy) with idempotency keys, `settlement_attempts`, retry with backoff → `MANUAL_REVIEW` (AS-03), `POST /settlements/{id}/retry`; commission calculation at acceptance (`CALCULATED`), **finalized only on `PAYOUT_COMPLETED`** (INV-5); listing-fee deduction; double-entry ledger postings for every leg with balanced journals and a deferred balance trigger (INV-6); compensating reversals only.
- **B builds:** bank funding screen (mark sent, generate OTP with **display-once "copy this now"** affordance, resends remaining); supplier OTP entry with attempts remaining and generic errors; settlement status timeline; payout tracking; failed-payout state with retry visibility (platform view).
- **Dependencies:** Phase 6.
- **Integration checkpoint:** live happy path to `FUNDED` via cross-party OTP; then the failure drill: dummy settlement adapter set to fail payout → `PAYOUT_FAILED` → automatic retries logged → manual retry succeeds → **assert exactly one payout leg in the ledger and journals balance**; commission status verifiably `CALCULATED` until payout completion, `FINALIZED` after.
- **Definition of done:** checkpoint + INV-5/6/10/13 tests + OTP brute-force test (6th attempt → `FAILED_MAX_ATTEMPTS`, audited).
- **Effort:** A 5–7 days; B 3–4 days.

### Phase 8 — Post-funding, cases, notifications (A) ∥ Post-funding + case UI (B)

- **Objective:** the lifecycle after money moves: maturity, overdue-unconfirmed discipline, payments, recourse, disputes, withdrawal, fraud, and evidence-grade notifications.
- **A builds:** maturity job on `TimeProvider` (reminders 30/14/7/due); `OVERDUE_UNCONFIRMED` on due-date pass (never straight to `OVERDUE` — ZM-PMT-008..011); bank confirm-status; buyer payment recording + reconciliation + derived balance (PA-06); partial → `PARTIALLY_PAID`, full → `PAID` → close; recourse (bank-only initiation, admin 403), repayments via settlement architecture, no automatic commission refund; disputes (automation paused while open — ZM-REC-013); withdrawal cases with configurable penalty calculation (recorded, not deducted) + admin decision + manual relisting flow (v3.1.0 + `relisting_requests` table per D-03); fraud review (freeze, funding stop, compliance notification, decision states); notification engine — versioned bilingual templates, EMAIL/WHATSAPP dummies + IN_PLATFORM + manual-call recording, full delivery evidence, the ZM-NOT-009 catalogue, buyer notification after confirmation (LT-14 wording, operational only); v3.1.0 notification inbox + case list/detail/decide endpoints.
- **B builds:** payment timeline with outstanding balance and overdue days; **`OVERDUE_UNCONFIRMED` phrased as "awaiting bank confirmation", never "defaulted"**; bank payment-report and confirm-status forms; recourse initiation (bank) and response (supplier); dispute views; platform case management (fraud, disputes, withdrawal, relisting approvals); notification inbox (in-platform channel).
- **Dependencies:** Phase 7; D-03/D-08/D-09 rulings.
- **Integration checkpoint:** with the demo time machine (built next phase — for this checkpoint, seed dates near maturity): a funded transaction passes due date → `OVERDUE_UNCONFIRMED` shown as awaiting confirmation → bank confirms `OVERDUE` → bank initiates recourse → supplier repays → `SETTLED` → transaction closed `RECOURSE_SETTLED`. Partial payment recalculates balance. A dispute pauses automation (maturity job skips the transaction, test-asserted). Buyer notification stored with delivery evidence.
- **Definition of done:** checkpoint + INV-7 (no hard delete) tests + supplier-view test asserting `bankInternalNotes` and evidence never serialize to supplier.
- **Effort:** A 6–8 days; B 5–7 days.

### Phase 9 — Admin, seed, demo, hardening (A + B converge)

- **Objective:** everything the demo needs, and nothing untested that the demo touches.
- **A builds:** admin settings/commission tiers/risk models/audit search finalized; **full seed**: 3 banks, 3 suppliers, 6 buyers, 12 invoices, users for every role, all **11 scenarios** positioned per the Seed-Data Specification (Part 2), each with a twin copy for demo fallback (Part 6); demo time machine (`/demo/time-travel`, server-side env guard returning 404 in production, every use audited + flagged simulated); complete invariant suite INV-1..13 green in CI; deployment runbook executed to the live stack.
- **B builds:** platform admin screens (settings, tiers, model versions, audit search); demo time-machine control (hidden unless the API reports it enabled — D-10); **full Arabic + RTL pass on every screen** (not a sample — brief §6), bidi checks for IBANs/Latin names in Arabic text; WCAG 2.1 AA pass (keyboard, focus, contrast, SR labels both languages); empty/loading/error states everywhere; performance pass (p95 < 2s on demo data).
- **Dependencies:** Phases 1–8.
- **Integration checkpoint:** the **full demo script (Part 6) executed twice end to end on the deployed stack — once in English, once in Arabic — by someone who didn't build it**, plus the reset script restoring seed state in under a minute.
- **Definition of done:** demo rehearsal sign-off; CI green including RLS, concurrency, and floor-leak suites; production-config check proving `/demo/time-travel` 404s when `demo_time_machine_enabled=false`.
- **Effort:** A 5–7 days; B 5–7 days.

## Part 3 — Work Split and Coordination Protocol

### 3.1 Ownership boundaries

| Path | Owner | The other agent may |
|---|---|---|
| `/apps/api/**` | **A** | Read only |
| `/services/ml/**` | **A** | Read only |
| `/db/**` (migrations, seed) | **A** | Read only |
| `/apps/web/**` | **B** | Read only |
| `docs/00–05_*` , `02_DATABASE_SCHEMA.sql`, `03_API_CONTRACT.yaml` | **Frozen — PO only** | Neither touches; defects go to `OPEN_QUESTIONS.md` |
| `docs/amendments/**` | PO (TL drafts) | Read; implement only after `DECISIONS.md` records approval |
| `docs/coordination/**` | Shared | Both append; neither rewrites the other's entries |
| Root config (`package.json` workspaces, CI, docker-compose, lockfiles) | **A** by default | B requests changes via daily log; A applies within a day, or B applies with an explicit note A acknowledges next session |
| `docs/specs/**` | Per Part 2 owner | Comment via coordination log |

**Neither agent may, without a `DECISIONS.md` entry:** modify frozen files; add non-additive migrations; change a response shape; introduce a new shared dependency at the repo root; rename an env var the other consumes.

### 3.2 Branch and merge strategy

- **Trunk-based.** `main` is protected (CI must pass). Each agent works on short-lived branches named `a/<phase>-<topic>` or `b/<phase>-<topic>` and merges to `main` at least daily — small merges, not phase-sized ones.
- Because ownership paths are disjoint, cross-agent merge conflicts should be **structurally impossible** outside root config. A conflict outside root config is itself an ownership violation — stop and log it.
- CI runs per-workspace: api tests, ml tests, web typecheck/tests, plus the cross-cutting suites (RLS, contract-conformance) nightly and on any `a/*` merge.
- **Contract conformance gate:** CI diffs Agent A's served `/docs-json` against `03_API_CONTRACT.yaml` (+ approved amendments). Divergence fails the build — this is the tripwire against silent contract drift.
- No rebases of shared history; merge commits are fine. Tag `phase-N-checkpoint` at every passed integration checkpoint so any regression bisects to a phase.

### 3.2a Phase completion reports (mandatory gate)

Every phase closes through `docs/completion/` (protocol: [../completion/README.md](../completion/README.md)):

1. Each agent writes `PHASE_<n>_AGENT_<A|B>.md` from the template **before starting its next phase's work** — delivered-vs-planned checklist, endpoints/screens with live status, tests added (invariants included), deviations and carry-overs, handoff notes.
2. The joint checkpoint is evidenced in `PHASE_<n>_CHECKPOINT.md` (steps, observed results, suite pass counts, git tag `phase-N-checkpoint`), written by the agent who ran it and countersigned by the other in its own report.
3. The product owner closes the phase with a `PHASE_<n> CLOSED` ruling in `DECISIONS.md`.
4. Separate files per agent per phase — the two sessions never write the same file, so reports can't collide any more than code can.

### 3.3 Daily handoff protocol

One append-only entry per agent per session in `docs/coordination/DAILY_LOG.md`:

```
## 2026-07-29 — Agent A (session 6)
LIVE: POST /transactions, PUT /transactions/{id}/invoice, GET /documents/{id}/extraction
CHANGED: none (contract untouched)
SEED: supplier-2 now has 2 draft invoices
BLOCKED ON: D-07 ruling (listing link)
NOTE FOR B: extraction mismatches array is empty until OCR job finishes (~5 s after upload)

## 2026-07-29 — Agent B (session 6)
DONE: invoice wizard steps 1–3 (mock), buyer block-states screen
SWAPPED TO LIVE: /buyers/search, /buyers/resolve — smoke passed
CONTRACT GAPS FOUND: none today
NEEDS FROM A: seeded buyer with SUSPENDED status for block-state testing
```

Rules: the log is the interface — agents do not depend on reading each other's code; every mock→live swap is recorded the day it happens; "NEEDS FROM A/B" items are answered in the counterpart's next entry or escalated.

### 3.4 Mock→live promotion (how B works while A builds)

1. **Source of truth for shapes is always the contract file**, never A's running server. B generates the typed client + MSW handlers from `03_API_CONTRACT.yaml` + approved amendments in Phase 1 and regenerates only when an amendment is approved.
2. B maintains `apps/web/lib/api/endpoint-status.ts` — a map of every operationId → `mock | live`, mirrored in `docs/coordination/ENDPOINT_STATUS.md`. MSW passes through to the real API for `live` entries; everything else stays mocked. The dev UI shows a badge listing which endpoints are mocked so demo-critical screens are never accidentally shown on mocks.
3. Mock data mirrors the **Seed-Data Specification identities** (same establishment numbers, same names) so the swap is visually diff-able.
4. Promotion: A posts LIVE in the daily log → B flips the map entry → B runs the screen's smoke flow → B confirms in its next log entry (or reports a contract-conformance bug, which goes to A, or a contract defect, which goes to the PO).
5. Endpoints never demoted silently: if a live endpoint breaks B, B flips it back to mock, logs it, and A treats it as a P1.
6. **Exit criterion:** by the Phase 9 rehearsal, the map contains zero `mock` entries on any demo-path endpoint (non-demo screens may degrade gracefully but must not error).

### 3.5 Ambiguity escalation

Order of authority per `00_START_HERE.md` §10: contract → schema → requirements → product owner.

1. Agent hits ambiguity → checks the three documents in order. If resolved, note the reading in the daily log (so the other agent inherits it).
2. Not resolved, or documents conflict → **stop that thread of work** (switch to another task — never block idle), append to `OPEN_QUESTIONS.md`: question, options considered, recommended answer, deadline needed by.
3. PO answers in `DECISIONS.md`. Both agents treat `DECISIONS.md` as binding addendum to the pack.
4. **Forbidden:** the two sessions resolving the same ambiguity independently; "temporary" workarounds that touch response shapes; B reading the SQL schema to reverse-engineer intent (per its brief — that situation is by definition a contract gap).

### 3.6 What Agent B does during the Phase 1 window (explicit, per constraint 2)

Design system; i18n scaffolding + RTL plumbing; message catalogs; codegen pipeline (client + MSW); Supabase Auth UI; static role-gated shells for all three portals; money components + lint rules; empty/loading/error patterns; the RTL checklist document. Zero live-data wiring until the Phase 1 checkpoint (`/auth/me` live) — exactly as `00_START_HERE.md` §5 prescribes.

## Part 5 — Test Strategy

### 5.1 The 13 invariants — one named CI test each

| Inv | Test (all run in CI; ★ = also in the nightly deep suite) |
|---|---|
| INV-1 | ★ Concurrency harness: 2 ACTIVE offers, N=20 repetitions of two parallel `POST /offers/{id}/accept` (different offers, same transaction) against real Postgres. Assert per run: exactly one 200, one 409, one `SELECTED`, others `NOT_SELECTED`, one snapshot, one `offer_selections` row. |
| INV-2 | Accept an offer, then (test hook) raise the floor above net → accept must 409. Boundary test at net == floor (must pass) and floor + 0.001 (must fail). |
| INV-3 | Offer with gross = outstanding + 0.001 rejected at create, revise, and accept. |
| INV-4 | After lock: direct SQL `UPDATE receivable_transactions SET locked_at = …` rejected by trigger; second accept 409; `locked_at` value byte-identical before/after. |
| INV-5 | Walk settlement to `PAYOUT_INITIATED`: commission still `CALCULATED`, `finalized_at` NULL. Complete payout: `FINALIZED`. Force `PAYOUT_FAILED`: never finalized. |
| INV-6 | ★ Property test: for every journal_id written by any suite, sum(DEBIT) = sum(CREDIT); plus a deliberate unbalanced insert asserting the deferred trigger rejects it. |
| INV-7 | `DELETE`/`UPDATE` against `audit_logs`, `ledger_entries`, and hard `DELETE` on invoices/offers/settlements as service role and as each persona → all rejected or no-op'd; row counts unchanged. |
| INV-8 | See 5.4 — the sentinel scan. |
| INV-9 | Paired-fixture test: identical transaction facts, run A all sources available, run B with GAM+ISTD `sourceAvailable=false`. Assert all five component scores identical, composite identical, `dataAvailabilityPct` strictly lower in B, and an information request raised iff the missing field is essential. |
| INV-10 | Matrix test: {OTP verified} × {settlement evidence} — only the (true, true) cell yields `FUNDED`; each single-condition cell stays `FUNDING_CONFIRMATION_PENDING`. |
| INV-11 | See 5.3 — RLS layer, not API. |
| INV-12 | Approve with `approver == creator` → 403 `SELF_APPROVAL_FORBIDDEN` at API **and** direct SQL UPDATE setting `approved_by = created_by` → CHECK violation. |
| INV-13 | ★ Force payout failure, run `POST /settlements/{id}/retry` 5× concurrently + let the job retry: dummy adapter's instruction log shows all calls carry the same idempotency key and exactly one payout executed; ledger has exactly one payout leg. |

### 5.2 Concurrency testing for atomic acceptance (beyond INV-1)

- Same-offer double-accept (both suppliers' clicks on one offer): one 200, one 409 — idempotency-key replay returns the original 200 body, not a second execution.
- Accept racing offer-withdrawal, offer-revision, and window-close jobs: no interleaving may produce `SELECTED` on a non-ACTIVE offer (run each pair 20×).
- Accept racing `reject-all`: exactly one of the two wins; state is never both locked and `ELIGIBLE`.
- Tooling: Jest + `Promise.all` against a real Postgres (Supabase local) — not mocks, not SQLite. The suite is tagged and required for any PR touching `selection/`.

### 5.3 Confidentiality at the RLS layer (INV-11), not only the API

- Dedicated suite `rls.spec` connects to Postgres **as the authenticated role with a forged-but-valid JWT** for each persona (bank-A ops user, bank-B ops user, supplier owner, platform admin), bypassing NestJS entirely (this is ZM-ARC-005's "policy must not pass because NestJS filtered first").
- For every tenant table (enumerated from the schema — the suite fails if a new tenant table appears without a policy entry): assert bank A's `SELECT *` returns zero of bank B's rows; specifically `bank_offers`, `bank_eligibility`, `bank_policy_filters`, `settlements`, `buyer_payments`, `funding_otps`.
- Assert bank A cannot infer competitor presence: `SELECT count(*) FROM bank_offers WHERE listing_id = X` as bank A returns only its own count (RLS filters before aggregate).
- Assert supplier can read ACTIVE offers on its listing but not `DRAFT`/`PENDING_INTERNAL_APPROVAL` ones (policy in frozen schema line: statuses ACTIVE/SELECTED/NOT_SELECTED — test the negative).
- Write-path RLS: bank A `INSERT`/`UPDATE` attempts on bank B's rows rejected.
- Run nightly and on any migration change.

### 5.4 Proving `minimumAcceptableAmount` never reaches a bank (INV-8)

Three independent layers, because this is a *absence* property:

1. **Sentinel scan (primary).** Seed every test-fixture floor with the sentinel `731937.731` (a value that cannot arise from any computation in the fixtures). The E2E suite records **every byte** of every response (headers included) received by any bank-persona session — success and error paths, including the deliberately-below-floor offer rejection — and greps for `731937` and for the key name `minimumAcceptableAmount`/`minimum_acceptable_amount`. Any hit fails the build. Also run against notification bodies and generated documents addressed to bank users.
2. **Serializer allow-list tests.** Bank-facing DTOs (`BankListingView`, bank's `Transaction` view, `Offer`, error payloads) are constructed by explicit field allow-lists; unit tests snapshot the exact key set so an added field is a visible diff.
3. **RLS column privilege.** After the D-02 fix, the suite from 5.3 asserts `SELECT minimum_acceptable_amount FROM receivable_transactions` as a bank user raises a permission error even on rows the bank may otherwise read.

Error-message discipline: the below-floor rejection test asserts the body contains exactly `code: OFFER_BELOW_SUPPLIER_REQUIREMENT` and no numeric content other than the offer's own figures.

### 5.5 Money precision

- **Backend:** decimal library (e.g. `decimal.js`/`big.js`) mandatory; ESLint rule banning `parseFloat`, `Number()` and arithmetic operators on money-typed values in `apps/api`; serializer test asserting every money field matches `^-?\d+\.\d{3}$`; DB round-trip test for values like `0.001`, `999999999999999.999`, and `1250.000` (no trailing-zero loss); the net-payout formula property-tested with 1 000 random component sets asserting DB CHECK, service computation, and API string agree exactly.
- **Adversarial cases:** commission on gross `100.005` at tier `1.25%` — rounding rule defined once (round half-up to 3 dp, documented in ARCHITECTURE.md), asserted identically in commission calc, offer validation, and settlement split (`chk_settlement_split` must hold with rounded values).
- **Frontend:** `MoneyDisplay`/`MoneyInput` unit tests: parse/format round-trips, Arabic locale renders JOD with 3 dp and correct digit set, inputs reject >3 dp; the same ESLint ban in `apps/web`; live net preview compared against server response in an integration test (must match to the digit or the UI shows the server value).

### 5.6 Bilingual and RTL pass

- **Coverage gate:** CI test that every message key exists in both `en.json` and `ar.json` (no fallback-to-English leaks) and no hardcoded user-facing strings (pseudo-locale build render check).
- **Per-screen RTL checklist** (from `RTL_CHECKLIST.md`), executed screen-by-screen in Phase 9 — every screen, not a sample: mirrored layout/nav/icons/table column order/progress direction; form flow; toasts and modals.
- **Bidi:** targeted Playwright assertions that IBANs, establishment numbers, and Latin company names embedded in Arabic sentences render LTR-isolated (visual snapshot + DOM `dir`/`bdi` checks) — brief §6 calls this out explicitly.
- **Playwright matrix:** the demo-path E2E suite runs in `en` and `ar`; visual snapshots reviewed for the most important screens (offer comparison, invoice wizard, funding OTP).
- **Semantics in both languages:** `OVERDUE_UNCONFIRMED` wording ("awaiting bank confirmation"), availability-neutral phrasing, and score disclaimer are asserted as translated strings, not just as keys.
- **Accessibility:** axe-core automated pass per screen + manual keyboard-navigation pass on the demo path in both directions (focus order mirrors correctly in RTL).

| Information required (SLA pause) | Data-only: an application sitting paused with visible clock + reason — one click to show on request | Supplier 3's application |
| Duplicate invoice | Data-only: review record + blocked submission visible; 20-second live repro on request (re-submit seeded invoice) | Supplier 2 |
| Fraud review | Data-only: frozen transaction + open compliance case in the platform portal | Transaction 9 |
| Failed payout + retry, no double pay | Data-only: settlement at `MANUAL_REVIEW` with attempt log; retry clickable on request | Transaction 10 |
| Full payment → PAID → CLOSED | Data-only: closed transaction timeline | Transaction 7 |
| Partial payment | Data-only: `PARTIALLY_PAID` with recalculated balance | Transaction 8 |
| Bank withdrawal after acceptance | Data-only: `WithdrawalCase` with penalty assessed + relisting decision | Transaction 11 |

Every data-only scenario must be reachable in ≤3 clicks from the relevant portal dashboard, because judges ask "what if the bank pulls out?" and the answer should be *showing*, not telling.

### 6.4 Time-machine usage

Exactly two jumps, scripted: **+40 days** (past due date of the funded transaction → maturity reminders, `OVERDUE_UNCONFIRMED`) and, if running the recourse live, **+7 days** (recourse escalation window). Each jump is audited and the audit entry is *shown* — it doubles as the audit-trail demo. The reset script clears `demo_time_offsets`. Rule: never jump backwards mid-demo (jobs don't un-run); the reset script is the only way back.

### 6.5 Must work vs. may be incomplete

**Must work (demo-blocking):** auth + org switching for all personas; invoice wizard incl. OCR pre-fill on *our seeded documents*; listing + fee display; eligibility; offer create/approve/self-approval-block; comparison screen; atomic accept; contract sign; OTP funding flow; settlement display + commission finalization; time machine; overdue flow; the seed + reset scripts; EN/AR toggle on the comparison screen; the deployed environment (with local fallback).

**May be incomplete without anyone noticing:** reports/analytics beyond simple dashboards (PA-03); WhatsApp adapter (email/in-platform evidence is what's shown); manual-call recording UI; admin settings *editing* (read view suffices); policy-filter editing beyond one filter; ML sophistication (rules-weighted score with the honest flag is acceptable — the *explanation display* is what judges see); accessibility polish beyond the demo path; Arabic on non-demo screens (must exist per requirements, but only comparison + one wizard screen are shown); document PDF rendering (PA-09).

### 6.6 Fallbacks if something breaks mid-demo

1. **Twin scenarios:** every live-demo entity has a pre-seeded twin one state earlier (e.g. second eligible invoice, second pair of approvable offers). A failed step = switch to the twin and continue the story, not restart.
2. **State-skip:** if a *transition* fails (e.g. OTP flow errors), the seed already contains the same story one state later (a `FUNDED` twin) — narrate the skipped step over its timeline/audit view, which shows the evidence of it having worked.
3. **Environment fallback:** local stack (`docker compose` + Supabase CLI) mirrors the deployed seed; switching is a browser-tab change rehearsed in advance. Decision rule: any infrastructure error twice in a row → switch environments, don't debug live.
4. **Reset:** `reset-demo` restores pristine seed in <1 min — usable between judge sessions, not mid-narrative.
5. **Last resort:** a rehearsal screen-recording of the full script, chaptered to match Part 6.2, played only for the broken segment.
6. **Preventive:** rehearse twice in Phase 9 (once by a non-builder); warm all services 15 min before; pre-open every tab; never live-type free text longer than a reference number (paste buffer prepared).

## Part 7 — Contract Defects (require product-owner ruling)

I did not silently redesign anything. Each item: what it is, what breaks, proposed fix. **Fix vehicle:** one approved amendment — API overlay **v3.1.0** (purely additive endpoints/fields) + migration **`0002`** (purely additive tables/grants) — recorded in `DECISIONS.md`. Severity: 🟥 blocks Phase 1, 🟧 blocks a specific phase, 🟨 clarification.

### 🟥 D-01 — `uq_active_invoice_fingerprint` is invalid PostgreSQL; the frozen schema does not load
- **Where:** `02_DATABASE_SCHEMA.sql` lines 429–434.
- **Problem:** partial-index predicates cannot contain subqueries (`WHERE transaction_id IN (SELECT …)` → PostgreSQL error `cannot use subquery in index predicate`). Running the file verbatim fails, so *every* Phase 1 migration is blocked.
- **Breaks:** the entire build; also ZM-VER-001/ZM-CON-017 (platform-wide duplicate block) if patched naively.
- **Proposed fix (behaviour-preserving):** add `invoices.is_active_fingerprint boolean NOT NULL DEFAULT true`, maintained by trigger on `receivable_transactions.state` (false when state ∈ {REJECTED, CANCELLED, CLOSED}); replace the index with `CREATE UNIQUE INDEX … ON invoices (fingerprint) WHERE is_active_fingerprint`. Additive column + trigger + corrected index; identical semantics.

### 🟥 D-02 — RLS lets an eligible bank read `minimum_acceptable_amount` by direct SQL
- **Where:** `tx_read` policy (schema lines 1126–1136) grants banks row-level SELECT on `receivable_transactions` for eligible listings; RLS is row-level, so **every column** of those rows — including the supplier's floor — is readable via Supabase's client API with a bank JWT. Violates INV-8/ZM-MKT-012 at exactly the layer the pack insists be independently safe (ZM-ARC-005, START_HERE §6 "test at the RLS layer").
- **Breaks:** the product's core confidentiality promise, silently.
- **Proposed fix:** `REVOKE SELECT (minimum_acceptable_amount) ON receivable_transactions FROM authenticated;` and route supplier/platform reads of the floor through the NestJS API (service role). Purely additive grant change; no table/API change. (Alternative: split the column to a supplier-private table — bigger change, not recommended under freeze.)

### 🟧 D-03 — `/admin/relisting-requests/{id}/approve` references an entity that doesn't exist anywhere
- **Where:** API line 1224; no `relisting_requests` table in the schema, no endpoint for a supplier to *create* a relisting request (ZM-MKT-016 requires a manual request + verification).
- **Breaks:** the bank-withdrawal seeded scenario and the manual-relisting workflow (Phase 8); the endpoint is unimplementable as specified.
- **Proposed fix:** migration `0002` adds `relisting_requests` (id, transaction_id, requested_by, status `REQUESTED|UNDER_REVIEW|APPROVED|DENIED`, verification jsonb per ZM-REC-018 checks, decided_by/at, timestamps); v3.1.0 adds `POST /transactions/{id}/relist-request` and `GET /admin/relisting-requests`.

### 🟧 D-04 — No way to create a supplier organization (bootstrap chicken-and-egg)
- **Where:** cross-cutting rule 1 requires `X-Organization-Id` on every request and membership in that org; a freshly-registered user has no organization, and there is no `POST /organizations` or registration endpoint. `POST /onboarding/applications` requires an `organization_id` that cannot exist yet.
- **Breaks:** the first five minutes of the product (Phase 2); Agent B's registration flow has no backend.
- **Proposed fix:** v3.1.0 adds `POST /onboarding/register` — exempt from the org-context header — which creates the SUPPLIER organization (status `ONBOARDING`), the membership with `SUPPLIER_OWNER`, and the draft application in one call, returning the new `organizationId` + `applicationId`. Document the exemption in the amendment.

### 🟧 D-05 — No list endpoint for supplier applications (reviewer queue and supplier resume both impossible)
- **Where:** only `GET /onboarding/applications/{id}` exists; the id is returned once at creation. A supplier logging in later cannot find their application; the platform "application review queue" screen (Agent B brief, Phase 8 §platform) has no data source.
- **Proposed fix:** v3.1.0 `GET /onboarding/applications` — role-scoped (supplier: own; reviewer: queue with status filter + pagination).

### 🟧 D-06 — No path from a transaction to its listing
- **Where:** `Transaction`/`TransactionSummary` schemas carry no `listingId`; `POST /transactions/{id}/listing` returns the Listing only at creation; there is no `GET /transactions/{id}/listing`. Supplier routes like `/offers/[listingId]` (Agent B brief) cannot be reached from the transaction list after a reload.
- **Proposed fix:** v3.1.0 `GET /transactions/{id}/listing` returning the current-round Listing (404 if none). (Adding `activeListingId` to `Transaction` would touch a frozen shape — additive but noisier; endpoint preferred.)

### 🟧 D-07 — No bank single-listing endpoint for the underwriting view
- **Where:** `BankListingView` is only returned inside the paginated `/marketplace/eligible` list. The underwriting screen (`listings/[id]` in the bank portal) has no fetch-one; `GET /listings/{id}` returns the thin `Listing` (and its `offerCount` field is supplier-only, complicating reuse).
- **Proposed fix:** v3.1.0 `GET /marketplace/listings/{id}` → `BankListingView` (403 unless the bank is eligible).

### 🟧 D-08 — No offer-collection endpoint: approval queue and "my offers" screens have no data source
- **Where:** offers are only reachable per-listing (`/listings/{id}/offers`) or by id. The bank portal's approval queue (`PENDING_INTERNAL_APPROVAL` across listings) and "my offers" list (both in Agent B's brief) cannot be built.
- **Proposed fix:** v3.1.0 `GET /offers?status=&page=&pageSize=` scoped to the active bank org (supplier/platform: 403 or scoped equivalents).

### 🟧 D-09 — Case-management surface is write-only
- **Where:** recourse, disputes, withdrawal, and fraud have creation endpoints but **no list, no get-by-id (except recourse's 201 body), and no decision endpoints** — yet the schema models decisions (`withdrawal_cases.decided_by`, `fraud_cases.status` transitions, `disputes.resolution_notes`) and the briefs specify bank `recourse/` and platform `cases/` screens plus the fraud/withdrawal seeded scenarios.
- **Proposed fix:** v3.1.0 adds: `GET /cases?type=&status=` (role-scoped), `GET /recourse/{id}`, `GET /disputes/{id}`, `GET /withdrawal-cases/{id}`, `GET /fraud-cases/{id}`, plus decision writes: `POST /withdrawal-cases/{id}/decide` (penalty + relisting eligibility), `POST /fraud-cases/{id}/decide` (CLEARED/RESTRICTED/SUSPENDED/BLACKLISTED/REPORTED), `POST /disputes/{id}/resolve`, `POST /recourse/{id}/status`.

### 🟧 D-10 — Nothing tells the frontend whether the demo time machine is enabled
- **Where:** Agent B must hide the control "unless the API reports it enabled"; `/admin/settings` is admin-only and `/demo/time-travel` 404s by design.
- **Proposed fix:** v3.1.0 adds optional `demo: { timeMachineEnabled: boolean, currentOffsetDays: integer }` to the `/auth/me` response (additive field, absent in production).

### 🟧 D-11 — No in-platform notification inbox endpoint
- **Where:** `IN_PLATFORM` is a required channel (ZM-NOT-001) with a `notifications` table, but no read/list/mark-read endpoints; the notification-driven UX (ZM-NOT-009: offers received, OTP pending, overdue…) has no surface.
- **Proposed fix:** v3.1.0 `GET /notifications?unread=` + `POST /notifications/{id}/read` scoped to the current user/context.

### 🟧 D-12 — Policy filters cannot be edited or deactivated; supplier cancellation has no endpoint
- **Where:** `/banks/policy-filters` has GET/POST only (brief: "policy filter configuration"); §16.8 cancellation/withdrawal rules (`SUBMITTED`/`OPEN_FOR_OFFERS` stages) have no endpoint.
- **Proposed fix:** v3.1.0 `PATCH /banks/policy-filters/{id}` (incl. `isActive:false`), and `POST /transactions/{id}/cancel` (stage-policy enforced server-side per §16.8).

### 🟨 D-13 — Invoice CHECK constraints forbid representing full repayment on the invoice row
- **Where:** `chk_outstanding (outstanding_amount = face_value - paid_amount)` + `chk_outstanding_pos (outstanding_amount > 0)`.
- **Issue:** if Agent A "recalculates the balance" (ZM-PMT-013) by updating `invoices.paid_amount`, a fully-paid invoice violates the constraint. Not a schema bug if — and only if — post-funding balance is derived from `buyer_payments` and the invoice row freezes at listing (PA-06). Needs to be *ruled*, not discovered.
- **Proposed fix:** no schema change; ratify PA-06 in `DECISIONS.md` so both agents implement the same reading (API's `/transactions/{id}/payments.outstandingAmount` = snapshot outstanding − Σ buyer_payments).

### 🟨 D-14 — Minor inconsistencies to note, no action proposed
- `funding_otps.status` defaults to `'SENT'` though the enum defines `PENDING_GENERATION` (rows are only created at generation, so harmless).
- `Money` pattern permits negatives while all DB constraints require positive (server validation covers it).
- `/transactions/{id}/funding/confirm` uses HTTP 401 for a wrong OTP (semantically odd next to auth-401s; frontend must branch on `code`, not status).
- No `webhook_events` table despite ZM-NFR-007 dedup; dummy adapters don't strictly need it — include in `0002` for adapter-swap readiness (additive, zero risk).
- `platform_settings` lacks keys for withdrawal-penalty policy and commission-refund-on-recourse policy (both `CONFIGURABLE` per LT-11/LT-12); add keys via seed (no schema change).

**If you find none of these persuasive:** D-01 is not a judgement call — the SQL does not execute. Everything else can in principle be deferred by descoping the affected screens, but D-02 would ship a silent confidentiality hole and D-04 makes onboarding undemonstrable, so I recommend approving D-01..D-12 as one amendment before Phase 2, with D-01/D-02 ruled before Agent A's first session.
