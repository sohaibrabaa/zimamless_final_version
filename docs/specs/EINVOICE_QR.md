# Electronic Invoice QR — payload schemas and parsing policy

**Owner:** Agent A · **Status:** Phase 3 deliverable
**Implements:** `ZM-DOC-007` (decode locally), `ZM-DOC-008` (compare against
OCR and supplier values), `ZM-DOC-009` (dummy validation adapter),
`ZM-DOC-010` (schema-driven, tolerant, degrade rather than guess)
**Code:** `services/ml/app/qr.py`, `services/ml/app/qr_schemas.py`,
`services/ml/app/einvoice.py`

---

## 1. The constraint this document exists to satisfy

> `ZM-DOC-010` The system **MUST NOT** hard-code an undocumented official QR
> payload structure. Parsing **MUST** be schema-driven and tolerant of format
> variation, degrading to "unparsed — manual review" rather than failing
> loudly or guessing.

Two things follow, and both are load-bearing:

1. **We do not have the official JoFotara payload specification**, and this
   document does not claim to reproduce it. Writing down a structure we
   inferred and labelling it "the Jordanian e-invoice QR format" is exactly
   the hard-coding the requirement forbids — it would look authoritative to
   the next reader and be wrong.

2. **Parsing is a registry of candidate schemas, tried in order.** Adding
   the real structure later is a new entry in `SCHEMAS`, not a rewrite. No
   existing schema's behaviour changes when one is added, which is what
   makes "replaceable without core changes" true rather than aspirational.

## 2. Supported schemas

Tried in the order listed. The first schema that detects, parses at least
one canonical field, and yields all of its own required fields wins.

| Schema | Version | Shape | Origin |
|---|---|---|---|
| `zimmamless-pipe-v1` | 1.0 | Pipe-delimited, `JO\|…` | **Ours.** The layout our seeded e-invoices carry. |
| `json` | 1.0 | JSON object with aliased keys | Generic. |
| `url-query` | 1.0 | Verification URL with query parameters | Generic. Several tax authorities publish a scan-to-verify link. |
| `tlv-base64` | 1.0 | Base64-wrapped tag/length/value triplets | The shape used by several regional e-invoicing systems. |

### 2.1 `zimmamless-pipe-v1` — our seeded format

```
JO|<einvoiceIdentifier>|<sellerEstNo>|<buyerEstNo>|<issueDate>|<total>|<tax>
```

Example, from `db/seed/einvoices/INV-2026-0001-alnoor-amman-retail.pdf`:

```
JO|JO-EINV-20000101-0001|20000101|30000201|2026-05-10|12354.000|1704.000
```

| Position | Canonical field | Format |
|---|---|---|
| 0 | *(literal `JO`)* | country marker, not a field |
| 1 | `einvoiceIdentifier` | `JO-EINV-<8 digits>-<4 digits>` |
| 2 | `sellerEstablishmentNumber` | 8 digits |
| 3 | `buyerEstablishmentNumber` | 8 digits |
| 4 | `issueDate` | `YYYY-MM-DD` |
| 5 | `faceValue` | 3-dp string |
| 6 | `taxAmount` | 3-dp string |

**Tolerance:** a payload carrying fewer trailing fields still yields the
fields it does carry. A short payload is incomplete, not invalid — refusing
it wholesale would discard the identifier we did read.

**This layout is ours.** It is generated by
`services/ml/tools/generate_einvoices.py` and read back by the same parser
the platform uses for any other invoice, so the seeded demo exercises the
real code path rather than a shortcut.

### 2.2 `tlv-base64`

Base64 of repeated `[tag:1 byte][length:1 byte][value:length bytes]`.

| Tag | Canonical field |
|---|---|
| 1 | `sellerName` |
| 2 | `sellerEstablishmentNumber` |
| 3 | `issueDate` |
| 4 | `faceValue` |
| 5 | `taxAmount` |
| 6 | `einvoiceIdentifier` |
| 7 | `buyerName` |
| 8 | `buyerEstablishmentNumber` |

Unknown tags are skipped, not treated as corruption. A truncated final
triplet keeps everything that parsed before it.

### 2.3 `json` and `url-query`

Keys are matched case- and punctuation-insensitively against an alias table
(`_JSON_ALIASES`), so `totalAmount`, `total_amount`, and `TOTAL` all reach
`faceValue`. A JSON payload with no recognisable keys is **not** claimed —
otherwise the schema would report a successful parse of an invoice with no
fields, which a caller cannot distinguish from a genuinely blank invoice.

## 3. Canonical field vocabulary

Every schema emits into the vocabulary in `services/ml/app/canonical.py`,
whose names match `InvoiceInput` in the frozen API contract exactly. That is
deliberate: OCR, QR, and the supplier's typed values are three independent
readings of one invoice, and the comparison in `ZM-DOC-008` is only possible
if all three use the same field names. A schema that emitted its own name
would silently never be compared — there is a test asserting none does.

Money is a **3-dp string** here as everywhere else in the system. A JSON
number would have lost precision before the API read it.

## 4. Outcomes, and what each one means

`validationStatus` on the extraction result:

| Status | Meaning | Reviewer action |
|---|---|---|
| `VALID` | A payload was read and a known schema understood it. | None. |
| `INVALID` | Understood, but the e-invoice validation adapter rejected the identifier. | Investigate. |
| `UNPARSED` | A payload **was** read; no schema understood it. | Manual review — the payload is preserved verbatim. |
| `UNAVAILABLE` | **No QR code was found on the document at all.** | None by itself. |

The last two are deliberately distinct, and the distinction is the same one
hard rule 7 draws for government sources: *"there was nothing to read"* and
*"there was something and we do not understand it"* are different facts.
Only the second is evidence of anything. Collapsing them into a single
"QR failed" would either send every QR-less invoice to manual review or hide
a genuinely unrecognised payload among them.

An unrecognised payload is always kept verbatim in `qr.rawOutput.payloads`.
The reviewer needs the actual bytes, not our failure to interpret them.

## 5. Local decoding (ZM-DOC-007)

Decoding is `cv2.QRCodeDetector` running in our own process. "Locally" is
the operative word in the requirement — an invoice carries the supplier's
counterparties, amounts and terms, and posting the page to a third-party
barcode service to read it would disclose all of that to obtain something
that takes a few milliseconds in-process.

Documents are rasterized before decoding (200 dpi for PDFs). Where a code
does not read at full colour, progressively more aggressive clean-ups are
tried — greyscale, Otsu threshold, adaptive threshold, 2× upscale — which
rescues faint print and uneven lighting on a photographed page. The first
success wins, so a clean document pays only for the first attempt.

## 6. The e-invoice validation adapter (ZM-DOC-009)

`services/ml/app/einvoice.py`. `EInvoiceValidator` is the seam; V3 ships
`DummyEInvoiceValidator`; a production implementation subclasses it and
nothing upstream changes.

The dummy validates the **shape** of an identifier and echoes what it was
asked about. It deliberately does not pretend to confirm that an invoice
exists in a national registry, because it cannot — a dummy that returned
`VALID` for everything would make the QR consistency check decorative while
appearing to work.

Failure-injection identifiers mirror the `9000000x` convention the
government dummy registry uses (`GOV_DUMMY_DATA.md` §5), so one debugging
habit covers both surfaces:

| Identifier | Outcome | `sourceAvailable` |
|---|---|---|
| `JO-EINV-90000001-0001` | `UNAVAILABLE` — the validator did not answer | `false` |
| `JO-EINV-90000002-0001` | `NOT_FOUND` — it answered, adversely | `true` |
| `JO-EINV-90000006-0001` | `INVALID` | `true` |

## 7. Seeded e-invoices

Generated by `services/ml/tools/generate_einvoices.py` into
`db/seed/einvoices/`. Output is byte-stable, so regenerating produces no
spurious diff, and `--check` fails if a file on disk has drifted from the
generator.

| File | What it exists to demonstrate |
|---|---|
| `INV-2026-0001-alnoor-amman-retail.pdf` | Happy path. S1 → B1, everything agrees. |
| `INV-2026-0002-alnoor-levant-mismatch.pdf` | **Deliberate mismatch** — the QR says `25000.000`, the page prints `24500.000`. |
| `INV-2026-0003-alnoor-aqaba-duplicate-a.pdf` | First half of the duplicate pair (S1). |
| `INV-2026-0003-petra-aqaba-duplicate-b.pdf` | Second half, same invoice data under S2 — must collide on fingerprint. |
| `INV-2026-0004-petra-amman-retail-past-due.pdf` | Past-due maturity, for AS-07. |

These are read back through the same rasterize-then-OCR path a scanned
upload takes. The platform is not reading a PDF text layer: it renders the
page to pixels and runs OCR over them, which is why the OCR confidence in
the extraction result is a real number below 1.0 and why misreadings are
possible at all.

## 8. Open items

- The **official JoFotara payload structure** is not implemented, because it
  is not available to us. When it is, it becomes a new entry in `SCHEMAS`
  and this table gains a row. Nothing else changes.
- The dummy validator confirms shape, not existence. Replacing it with a
  real validation service is the `EInvoiceValidator` subclass described in
  §6.
