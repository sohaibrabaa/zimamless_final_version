# ML Design — Trust Score, risk model, and the availability rule

**Owner:** Agent A · **Phase:** 4 · **Status:** implemented
**Code:** `services/ml/app/risk/`, `apps/api/src/modules/risk/`

This document states what the model is, what it was trained on, what it can
and cannot support, and how the architecture makes ZM-RSK-005 (INV-9) hold as
a property rather than as a promise.

---

## 1. The honest limitations, first

Everything in §2–§8 should be read against these four facts. They are stated
first rather than in a closing caveat because a reader who stops early should
still leave with them.

1. **The training data is synthetic.** There is no historical Jordanian
   receivables-default dataset available to this project. The data is
   generated by `services/ml/app/risk/synthetic.py` from an explicit
   process written out in §4. Every artifact records `"synthetic": true`,
   every prediction carries the flag, and every score carries the reason code
   `INFO_SYNTHETIC_TRAINING_DATA`.
2. **Therefore the reported metrics measure recovery, not prediction.** An AUC
   of 0.91 says the trainer successfully recovered the relationships the
   generator encoded. It says nothing about real-world default prediction, and
   no report, screen, or demo narration should imply otherwise (ZM-RSK-016).
3. **This is not fraud forensics.** The integrity checks are deterministic
   (hash comparison, fingerprint collision, OCR/QR consistency) and the model
   contributes anomaly *estimation*. ZM-RSK-018 forbids presenting any of it
   as production-grade forensic detection, and it is not.
4. **The score is decision support only** (ZM-RSK-001). The disclaimer is
   returned in both languages with every assessment and is not suppressible.

## 2. Architecture: two separated concerns

ZM-RSK-015 requires deterministic rules and a trained model to be separate,
with the model unable to override a hard blocker. The separation is enforced
by the order of operations, not by policy:

```
  facts ──► five components ──► rules composite ──► blend with model ──► CAP
   │              │                                        │             │
   │              └─► dataAvailabilityPct (separate)        │             │
   │                                                        │             │
   └─► hard blockers ───────────────────────────────────────┴────────────►┘
```

`capForBlockers` runs **last**. There is no model output that lifts a blocked
transaction out of `CRITICAL`, because the cap is applied to whatever the
blend produced. The named test is
`scoring.spec.ts › ZM-RSK-015 › caps a blocked transaction below the CRITICAL threshold`.

| Layer | File | Responsibility |
|---|---|---|
| Facts | `apps/api/src/modules/risk/facts.ts` | Availability as a *type*, not a value |
| Rules | `rules-engine.ts` | Hard eligibility and fraud blockers |
| Scoring | `scoring.ts` | Five components, composite, band, availability |
| Codes | `reason-codes.ts` | The structured vocabulary shared with the UI |
| Model client | `risk-model-client.service.ts` | Calls the ML service; never throws |
| Versions | `risk-models.service.ts` | Create-never-edit lifecycle |
| Orchestration | `risk.service.ts` | Gathers, scores, persists, presents |
| Model | `services/ml/app/risk/` | Features, data, training, inference |

## 3. How INV-9 is made structural

> ZM-RSK-005: government downtime, an unavailable source, or a field the
> source does not publish **must never reduce the Trust Score**.

Three independent mechanisms, at three layers. Any one of them would mostly
work; together they make the failure mode hard to reintroduce by accident.

### 3.1 The type system (`facts.ts`)

A fact is `Known<T>` or `Unavailable`. An `Unavailable` has **no `value`
property**, so no scoring rule can read one. There is deliberately no
`valueOr(default)` helper — a default is precisely the thing that converts an
absence into a score.

`Unavailable` also carries *why*: `SOURCE_UNAVAILABLE` (the source did not
answer) or `NOT_PUBLISHED` (it answered but does not publish this field).
ZM-RSK-008 requires that distinction to survive to the UI, and it emits
different INFO codes accordingly.

### 3.2 The arithmetic (`scoring.ts`)

Each component is a list of weighted signals, each either evaluated or
unavailable:

```
  component = 100 × Σ(weight × points | EVALUATED) / Σ(weight | EVALUATED)
```

The denominator is the **evaluated** weight. An unavailable signal leaves both
the numerator and the denominator, so it cannot move the result. Scoring it as
zero — the obvious-looking implementation — would keep it in the denominator
and silently penalise the supplier for someone else's outage.

The composite renormalises the same way over components that produced a score,
so a component that could not be scored at all does not drag it toward zero.

Availability is computed by a **separate function over the same signals**:

```
  dataAvailabilityPct = 100 × Σ(weight | EVALUATED) / Σ(weight | ALL)
```

Two functions, no shared code path, so the paired-fixture test can assert that
one moved and the other did not.

### 3.3 The feature set (`features.py`)

The model is the layer where the requirement is hardest to enforce after the
fact. If `registry_status_known` were a feature, the model would learn
"unknown registry ⇒ riskier" — correctly, from the data, and in violation of
the requirement. No downstream clamp undoes a model that has internalised the
correlation.

So the correlation is made **unlearnable**: every feature is derived from a
fact the platform holds in its own database. Not one comes from a government
registry or any source that can be down. The model's output is therefore
provably invariant to source availability, by omission.

### 3.4 What *does* reduce the score

ZM-RSK-007, and the distinction the whole product rests on:

| Situation | Effect |
|---|---|
| Registry answers "STRUCK_OFF" | Hard blocker, band CRITICAL |
| Registry answers "UNDER_LIQUIDATION" | Risk factor, lowers `buyerProfile` (LT-02: review, not refusal) |
| Registry does not answer | **No score change.** Lowers availability, emits `INFO_GOVERNMENT_SOURCE_UNAVAILABLE` |
| Registry answers but omits the field | **No score change.** Lowers availability, emits `INFO_FIELD_NOT_PUBLISHED` |

Named test:
`scoring.spec.ts › adverse findings DO reduce the score › separates "registry says struck off" from "registry did not answer"`.

## 4. The training data

`services/ml/app/risk/synthetic.py`, seed `20260723`, 6 000 samples.

Marginals are chosen to resemble the platform's own population rather than a
textbook: most invoices are clean, most suppliers have short histories,
disputes are rare.

| Feature | Distribution |
|---|---|
| `tenor_days` | Gamma(4, 22), clipped to 1–365 |
| `face_value` | Lognormal(9, 1), clipped to 100–5 000 000 |
| `tax_ratio` | Normal(0.16, 0.02), clipped to 0–0.30 |
| `completeness_ratio` | Beta(9, 1.2) |
| `duplicate_collision` | Bernoulli(0.04) |
| `electronic_invoice_attached` | Bernoulli(0.94) |
| `partially_paid` | Bernoulli(0.12) |
| `prior_submitted_count` | Poisson(3.0) |
| `dispute_count` | Poisson(0.18) |
| `duplicate_referral_count` | Poisson(0.10) |
| `recourse_count` | Poisson(0.08) |

Labels are drawn from a logistic ground truth over the **standardized**
features (so the intercept controls the base rate independently of each
feature's scale), with weights in `TRUE_WEIGHTS` and intercept −1.15. The
resulting positive rate is 0.317.

Determinism is a hard requirement: `numpy.random.Generator(PCG64(seed))`, no
global RNG, no shuffling. The same seed produces byte-identical data.

## 5. The model

Logistic regression, written out in `model.py` rather than imported from
scikit-learn, for three reasons that point the same way:

1. **Explainability is the requirement.** In standardized space, feature *j*'s
   contribution to a prediction is exactly `coef[j] × z[j]` — a signed
   log-odds term. With the intercept, the contributions **sum to the
   prediction**. Nothing is approximated or sampled, and there is no separate
   explainer library whose output could disagree with the model. There is a
   test asserting the reconstruction holds to 1e-6.
2. **Determinism.** Full-batch gradient descent from a zero start: no random
   initialisation, no shuffling, no early-stopping heuristic.
3. **The pure-wheel constraint** this service already lives under (see
   `requirements.txt`). scikit-learn is a large dependency for ~60 lines of
   arithmetic.

Training: 4 000 iterations, learning rate 0.35, L2 1e-3, deterministic 75/25
train/test split by slice (the data is generated in random order, so a tail
slice is as unbiased as shuffling and adds no second seed).

### Recorded metrics (ZM-RSK-017)

Artifact `risk-logreg-1.0+seed20260723`, stored in
`risk_model_versions.training_metrics` so a historical score stays explainable
even if the ML service is rebuilt or retired.

| Metric | Value |
|---|---|
| Samples (train / test) | 6 000 (4 500 / 1 500) |
| Positive rate | 0.3165 |
| Accuracy | 0.851 |
| ROC AUC | 0.910 |
| Brier score | 0.107 |
| Log loss | 0.343 |

AUC is computed exactly via the rank-sum identity with tied scores averaged,
not trapezoid-approximated.

**Read metric 2 in §1 before quoting any of these numbers.**

### Reproducibility

`py -m tools.train_risk_model --check` retrains and asserts the result is
byte-identical to the committed artifact. It exists for the same reason the
e-invoice generator has one: a "deterministic" pipeline nobody re-runs is a
claim, not a property. If a numpy upgrade changes a reduction order, the check
fails rather than the model quietly drifting from its recorded metrics.

## 6. How the model enters the score

The model speaks in P(goes bad); the score speaks in trust, so the probability
is inverted and blended:

```
  blended = (1 − w) × rulesComposite + w × 100(1 − p),  w = 0.25
```

The weight is a deliberate minority, stored on the model version as
`weights.ml` and adjustable by creating a new version. The reasoning: the
model is trained on synthetic data and its predictive validity on real
receivables is unestablished, so letting it dominate a number a bank might
rely on would overstate what it knows. The rules carry the decision; the model
adjusts it.

Then the blocker cap is applied — see §2.

### Fallback (ZM-RSK-017)

The client never throws. A transport failure, a non-200, or a 200 that reports
no usable artifact all produce the same degraded shape, so the caller has one
code path and cannot forget to handle the down case.

When the model does not answer: `mlUsed=false`, `mlFallbackReason` is
populated with the specific reason, the reason code
`INFO_ML_UNAVAILABLE` is recorded, and the score is produced by rules alone.
The degradation is **visible to every audience including banks** — a banker
relying on a score is entitled to know the model did not run.

This is not theoretical: during Phase 4 development the journey suite caught a
stale ML process serving an older build without the risk route, and reported
it as a visible rules-only fallback rather than an error.

## 7. Versioning (ZM-RSK-009..011)

- **Create, never edit.** `RiskModelsService` has no update method and no
  route reaches one. A score's meaning is fixed by the weights that produced
  it; editing an active version in place would silently redefine every
  historical score without touching a single `risk_assessments` row.
- **Exactly one active version**, enforced by `uq_one_active_risk_model`, a
  partial unique index. Deactivate and activate share one transaction.
- **Activation is audited** with actor, timestamp and a required rationale.
- **Historical scores never change.** Named test:
  `phase4-risk-versions.integration.spec.ts › keeps a historical assessment
  unchanged when a new version is activated (ZM-RSK-010)` — it activates a new
  version with radically different weights *and thresholds* and asserts the
  stored assessment keeps its score, its band, and its version pointer.

## 8. Bank-facing disclosure (ZM-RSK-012 / 013)

A bank receives: composite score, band, five components, `dataAvailabilityPct`,
positive factors, risk factors, reason codes, model version, calculation date,
`mlUsed` (+ reason when false), and the disclaimer.

A bank does **not** receive: weights, coefficients, means, standard
deviations, the raw model probability, or per-feature contributions. The
response is built from an explicit allow-list — never by spreading the row —
so a column added later cannot leak by default. This is the same discipline
INV-8 established on the transaction payload.

Two layers, because one is not enough:

- **API:** `RiskService.describe()` allow-list, with a named test asserting
  the serialized bank payload contains none of the forbidden keys.
- **Database:** migration `0006` revokes `SELECT` on
  `risk_model_versions.weights` and `.training_metrics` from `authenticated`.
  Migration `0003` had left the whole row world-readable on the reasoning that
  "weights are not secret either — explainability is a requirement", which
  conflated two different things: explainability is discharged by per-
  prediction factors and reason codes, not by handing over the weight vector.
  As written, any authenticated bank user could have read the entire scoring
  configuration straight from the Supabase client, bypassing the API's
  allow-list. `db:verify` now checks both columns.

Note the technique, which is not obvious: a bare `REVOKE SELECT (col)` does
nothing while a table-level SELECT grant exists, because table-level
privileges satisfy a column read on their own. Postgres accepts the statement
and changes nothing. The table grant must be dropped and re-issued column by
column — the same pattern D-02 uses for the supplier floor.

## 9. What would make this real

Named so the gap is explicit rather than implied:

1. **Real labelled outcomes.** Every metric in §5 is replaceable the day the
   platform has a few hundred settled receivables with known outcomes. The
   feature vector and the training script do not change; only `synthetic.py`
   is retired and the `synthetic` flag becomes false.
2. **Calibration.** The current model is not calibrated against a real base
   rate, so the probability should be read as an ordering, not a likelihood.
3. **Buyer-side features.** The strongest real-world predictors are about the
   debtor, and today the buyer contributes through the rules component only —
   deliberately, since registry-sourced features would breach §3.3. A real
   deployment would need availability-invariant encodings (for example,
   imputation with an explicit availability-independent prior) rather than
   simply adding the fields.
4. **Adverse-action explanations.** ZM-RSK-012 covers what a bank sees. If the
   score ever gates a supplier's access, the supplier needs its own
   explanation path.
