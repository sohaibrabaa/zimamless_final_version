## 4. Deviations, and the defects found by running things

### 4.1 The Phase 0 "Python is not installed" carry-over was wrong

Carried since Phase 0 and repeated in the Phase 3 kickoff. It is not true:
`python` resolves to a Microsoft Store stub, but the **`py` launcher runs
Python 3.13.3**, and `py -m ensurepip` restores pip. The carry-over had
survived two phases on the strength of one command that fails in a
misleading way.

The service is therefore real, and so is the OCR. Every dependency installs
as a pure wheel with no native toolchain and no separately-installed binary
— **RapidOCR ships its ONNX models inside the wheel**, which is why there is
no Tesseract prerequisite for CI or the demo machine. `pip install -r
requirements.txt` is the whole setup on every platform.

The platform rasterizes each page at 200 dpi and reads **pixels**. It does
not read the PDF text layer, which would have been faster and exact — and
would have meant the platform reads only invoices that happen to have one,
silently reading nothing from the scan or phone photo a supplier with a
paper invoice actually uploads.

### 4.2 Real OCR immediately found two defects that unit tests had not

Both were substring matches that should have been word matches, and both
produced *plausible* output rather than an obvious failure:

- **Every Al-Noor invoice silently lost its seller name.** The reject token
  `"no."` canonicalizes to `no`, which is a substring of `noor`. Company
  names are full of short tokens that appear inside longer ones.
- **A totals line could be read as the buyer's name.** The buyer label `to`
  prefix-matches `total 12354 000 jod`. This one had not surfaced yet only
  because the buyer line happens to be printed above the totals.

Both are now word-boundary matches with named regression tests. A third —
the goods description being pre-filled with the word "Amount" from a table
header — was fixed by refusing to guess at all rather than by a cleverer
heuristic (ZM-DOC-010).

### 4.3 Every date the API emitted was one day early

The worst defect of the session, and invisible to typecheck and to every
unit test.

A Postgres `date` arrives through node-postgres as a JS `Date` at **local**
midnight. `toISOString().slice(0,10)` then moves it to the previous day in
any timezone ahead of UTC — which includes Asia/Amman, the only timezone
this product runs in.

It surfaced as an OCR-vs-invoice mismatch on a document whose dates had been
read perfectly: the page said `2026-05-10`, OCR said `2026-05-10`, and the
platform believed the supplier had entered `2026-05-09`. The submission
landed in `UNDER_REVIEW` instead of `ELIGIBLE`, and the reason was our own
serialization.

It also fed the fingerprint, so two genuinely identical invoices submitted
either side of a timezone boundary would have fingerprinted differently.

Fixed by selecting every `date` column as `::text` and typing it as a
string. **A calendar date has no timezone, so it never becomes a `Date`
object.** Applied to invoices (issue/due) and buyers (registration date).

### 4.4 The duplicate check was defeated by the index meant to enforce it

`uq_active_invoice_fingerprint` covers every non-terminal state **including
`DRAFT`**. Writing the real fingerprint while the supplier was still editing
meant the *second* draft of a duplicate hit a raw unique violation on
`PUT /transactions/{id}/buyer` — a 500, at which point the service's own
duplicate check never ran and `submit` saw no collision at all. The
protection and the thing it protected cancelled out.

Now a draft carries `draft:<transaction-id>`, unique by construction, and
the real fingerprint is computed and written once inside submit's duplicate
check. That puts the refusal exactly where the contract declares it (409 on
submit, with a review record) and leaves the unique index as what it should
be: the backstop for two submissions racing, handled explicitly as a 23505
that becomes the same 409.

### 4.5 `POST /documents/{id}/finalize` was designed, built, and then removed

The upload goes straight from the browser to Supabase Storage, so the API
never sees the bytes and cannot hash them at upload time. The obvious design
is a finalize call. The conformance gate refused it — correctly: it is in
neither the frozen contract nor the approved overlay, Agent B could not
generate it, and hard rule 1 forbids inventing an endpoint without a ruling.

Rather than file a question and stall the phase, hashing and extraction were
made **lazy**, triggered by the first operation that genuinely needs the
file's content (reading its extraction, or submitting). No contract change,
and better than the explicit call regardless: a client that forgot to
finalize would otherwise leave an unhashed document that silently failed its
integrity check later.

### 4.6 A pre-existing break that made the migration runner unusable

Not mine, but it blocked the phase and is now fixed.

`node db/tools/migrate.mjs` refused every operation: five applied migrations
"modified on disk". Two separate causes.

- **Line endings.** `.gitattributes` mandates `eol=lf`, but `core.autocrlf`
  was `true` in this checkout, so the working tree held CRLF while the index
  held LF. Three of the five checksums came right the moment the tree was
  re-materialized. (This is the same class of failure the `.gitattributes`
  comment was added to prevent.)
- **`0001` and `0002` were recorded from bytes that no longer exist
  anywhere** — not the committed files, not the `_manual` bundle, in either
  line ending. They were applied on 2026-07-22 from a working copy that was
  edited before being committed.

`db:verify` and the frozen-schema drift check both passed throughout, so the
database itself was correct; the bookkeeping was not. Added
`migrate.mjs --rebaseline <name>`: re-records an already-applied migration's
checksum, refuses `all`, refuses a migration that is not already recorded,
and prints both checksums. Deliberately narrower than `--baseline` so
"overwrite history" cannot be done by reflex.

### Design decisions worth recording

- **The fingerprint keys on national establishment numbers, not internal
  buyer ids**, so it survives a buyer record being merged or recreated. An
  internal id would let a duplicate through whenever two suppliers happened
  to resolve to different rows for one company.
- **Normalization is deliberately narrow**: case, spacing and separator
  punctuation are stripped from identifiers because they carry no meaning;
  amounts keep full 3-dp precision. Too strict and a fraudster evades by
  retyping; too loose and two genuine invoices to the same buyer on the same
  day for the same amount collide, blocking an honest supplier and opening a
  fraud case against them.
- **`NOT_APPLICABLE` is not `PASS`.** A check that could not run because its
  input is absent must not report success — a missing QR code is not
  evidence that the QR agreed with anything.
- **"No QR on the page" and "a QR we could not read" stay distinct**, the
  same distinction hard rule 7 draws for government sources. Only the second
  warrants a human.
- **Only a file-integrity failure escalates to `FRAUD_REVIEW`.** ZM-VER-002
  is explicit that a failed check is not proof of fraud; treating an OCR
  mismatch as fraud would accuse a supplier over a smudged scan.
- **The ML service holds no credentials and never touches the database.** A
  second copy of the authorization rule there would be a second place for it
  to be wrong; only there would be worse, since a service with no notion of
  organizations cannot enforce a rule about them.
- **The 409 for a duplicate names no counterparty.** Telling the submitter
  whose invoice they collided with would tell a fraudster exactly which of
  their attempts landed first.
- **`0300_seed_phase3.sql` deliberately seeds no invoices or documents.**
  A hand-written invoice row would be a fiction — its fingerprint, its
  extraction rows and its stored object are all produced by code paths that
  must actually run for the row to mean anything. The e-invoice PDFs are the
  Phase 3 seed.

  policy, every ML module) have real unit coverage. The *services* are
  proven by the 36-check live run, which is reproducible by hand but is not
  in CI — a regression in, say, the lazy-finalization path would not be
  caught automatically. Listed as a carry-over rather than claimed as
  covered.
- **OCR accuracy is demonstrated on documents we generated.** The extraction
  is genuine — real rasterization, real pixels, and it makes real mistakes
  (it reads "Identifier" as "ldentifier") — but the layouts are ours. A
  supplier's real invoice will have a layout the label schema has not seen,
  and the honest expectation is that it degrades to partial pre-fill, which
  is the designed behaviour rather than a failure.
- **The Phase 3 tables now have live-run residue**, exactly as Phase 2's
  queue did: transactions `ZM-1004`+ were created by the verification script
  and have random ids. They are real rows, not seeded fixtures. Named here
  so the distinction is not lost the way it was in the Phase 2 report.

most affect the wizard:

1. **`POST /buyers/resolve` returns 200, not 201**, and requires
   `confirmedByUser: true`. `SUSPENDED`/`STRUCK_OFF` are 409 `BUYER_BLOCKED`;

   adverse. `UNPARSED` means a code was read and no schema recognised it —
   that one is manual review. Do not collapse them.

   the 422 refusing an excessive floor deliberately does not echo the floor
   back. Do not reconstruct it from the error.
5. **All eight declarations must be true**; a false one is **422** with

   `details.reviewReference` for the blocked-screen's review reference. It
   deliberately tells you nothing about the other party.
7. **Money is a 3-dp string on the way in.** `"12354"` is rejected;
